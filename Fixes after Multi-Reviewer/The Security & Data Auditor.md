# Security & Data Audit: MBTI Theory Quiz

**Auditor role:** Security & Data Auditor  
**Scope:** Password hashing, file obfuscation, safe file handling, data processing  
**Reference:** SPEC.md (data format, file structure, password handling requirements)  
**File under review:** `quiz.py` (701 lines), plus all data files

---

## Findings

### Password Hashing & Storage

**1. [PASS] Passwords are hashed with PBKDF2-HMAC-SHA256 and a random salt — `quiz.py` lines 113–118**

`hash_password()` uses `hashlib.pbkdf2_hmac("sha256", password.encode(), salt, 100_000)` with a cryptographically random 16-byte salt generated by `os.urandom(16)`. The iteration count of 100,000 is within OWASP's recommended minimum. The stored format is `salt_hex:hash_hex`, which properly separates the salt from the derived key. This exceeds the SPEC.md requirement of "hashed using hashlib" and follows modern best practices.

**2. [WARN] Legacy unsalted SHA-256 fallback in password verification — `quiz.py` line 124**

`verify_password()` contains a backwards-compatibility path: if the stored hash does not contain a `:` separator, it falls back to a plain `hashlib.sha256(password.encode()).hexdigest()` comparison. Unsalted SHA-256 is vulnerable to rainbow-table and precomputation attacks. Any user accounts created before the PBKDF2 upgrade remain protected only by this weaker scheme. There is no migration path that automatically re-hashes legacy passwords to the stronger format upon successful login.

**3. [WARN] Passwords are entered via `input()` instead of `getpass` — `quiz.py` lines 167, 199**

Both `create_account()` (line 167) and `login()` (line 199) read passwords with `input()`, which echoes the password in plaintext on the terminal. The standard library's `getpass.getpass()` suppresses echo and is the expected approach for terminal password entry. While SPEC.md does not explicitly require hidden input, this is a basic security hygiene concern.

**4. [PASS] Plaintext passwords are never written to disk — `quiz.py` lines 179–185**

The raw password string from `input()` is immediately passed to `hash_password()` and only the returned hash string is stored in the user dict at line 183. The password variable goes out of scope when `create_account()` returns. No logging or temporary file writes expose the plaintext.

**5. [PASS] Login brute-force is rate-limited — `quiz.py` lines 38, 198–208**

`MAX_LOGIN_ATTEMPTS = 3` (line 38) limits consecutive password guesses per login session. After 3 failures, the user is returned to the main menu. This satisfies a basic lockout mechanism, though it is per-session only (restarting the login flow resets the counter).

### Data File Obfuscation (users.dat / scores.dat)

**6. [PASS] Binary files use zlib-compressed JSON, not pickle — `quiz.py` lines 63–77**

`_load_dat()` and `_save_dat()` serialize data as JSON, then compress with `zlib`. This produces non-human-readable `.dat` files that satisfy SPEC.md's requirement ("Non-human-readable format"), while completely avoiding the arbitrary code execution risk of `pickle.load()`. This is a significant security improvement over the pickle approach described in the original SPEC.md suggestion.

**7. [PASS] scores.dat is obfuscated — `quiz.py` lines 88–93**

Score data flows through the same `_save_dat()` / `_load_dat()` helpers, so `scores.dat` is zlib-compressed binary JSON. A user cannot casually open the file in a text editor and read or edit scores. This meets SPEC.md's requirement that scores.dat be "relatively secure" and "non-human-readable."

**8. [WARN] Obfuscation is compression, not encryption — `quiz.py` line 77**

`zlib.compress()` is a compression algorithm, not an encryption algorithm. Anyone with basic Python knowledge can decompress and read the contents with `zlib.decompress(open("users.dat","rb").read())`. While SPEC.md only requires "non-human-readable" (which this satisfies), the data is not protected against a motivated attacker. Password hashes inside the file are still hashed, so this is primarily a concern for score data integrity.

**9. [PASS] Corrupted .dat files are handled gracefully — `quiz.py` lines 68–71**

`_load_dat()` catches `zlib.error` and `json.JSONDecodeError` separately from `FileNotFoundError`, returning an empty `{}` in all failure cases. If a `.dat` file is truncated or corrupted, the application does not crash — it starts fresh, which aligns with SPEC.md's requirement: "if users.dat or scores.dat gets deleted mid-use, the app catches the missing file error and just makes a new empty save file."

### Safe File Handling

**10. [PASS] All file reads use `with open(...)` context managers — `quiz.py` lines 47, 57, 66, 98**

Every file read in the codebase uses a `with` statement:
- `load_questions()` — line 47: `with open(QUESTIONS_FILE, "r") as f:`
- `load_tips()` — line 57: `with open(TIPS_FILE, "r") as f:`
- `_load_dat()` — line 66: `with open(filepath, "rb") as f:`
- `load_feedback()` — line 98: `with open(FEEDBACK_FILE, "r") as f:`

Files are guaranteed to be closed even if an exception occurs mid-read.

**11. [PASS] All file writes use `with open(...)` context managers — `quiz.py` lines 76, 105**

Every file write in the codebase uses a `with` statement:
- `_save_dat()` — line 76: `with open(filepath, "wb") as f:`
- `save_feedback()` — line 105: `with open(FEEDBACK_FILE, "w") as f:`

File handles are properly released after writing. No manual `.open()` / `.close()` pairs exist anywhere in the code.

**12. [WARN] Writes are not atomic — `quiz.py` lines 76, 105**

Both `_save_dat()` and `save_feedback()` write directly to the target file. If the process is killed or the system crashes mid-write (e.g., power loss during `f.write(zlib.compress(...))`), the file could be left truncated or empty. The safe pattern is to write to a temporary file in the same directory and then `os.replace()` it onto the target. For a homework-scale application this is low-risk, but it is a known data-corruption vector in production systems.

**13. [PASS] Missing questions.json produces a clean exit — `quiz.py` lines 46–52**

`load_questions()` catches `FileNotFoundError`, `json.JSONDecodeError`, and `KeyError`, prints `"Oops, can't find the questions file!"`, and calls `sys.exit(1)`. This exactly matches SPEC.md's acceptance criterion: "Running the app with a missing questions.json prints an error and exits with code 1."

**14. [PASS] Missing tips.json does not crash the app — `quiz.py` lines 55–60**

`load_tips()` returns an empty `{}` on `FileNotFoundError` or `json.JSONDecodeError`. Downstream code at lines 323–325 checks `if user_mbti in tips and tips[user_mbti]` before accessing, so a missing tips file simply means no tip is shown. The app continues normally.

**15. [PASS] Missing feedback.json does not crash the app — `quiz.py` lines 96–101**

`load_feedback()` returns `{}` on failure. All feedback access goes through `.get()` with defaults (e.g., line 413: `user_feedback.get(section, 3)`), so no `KeyError` can occur.

### Data Processing & Input Safety

**16. [PASS] Integer inputs are validated with range bounds — `quiz.py` lines 131–144**

`get_int_input()` catches `ValueError` for non-numeric input and enforces `min_val` / `max_val` bounds. Every menu selection and quiz answer in the app routes through this helper, preventing index-out-of-range errors and type crashes. This satisfies SPEC.md: "if a user enters an invalid menu option like typing letters instead of a number, the app won't crash."

**17. [PASS] Short answer comparison is case-insensitive — `quiz.py` lines 301–302**

`answer.lower() == q["answer"].lower()` normalizes both sides before comparison. This matches SPEC.md: "short answer questions should accept both 'intj' and 'INTJ'."

**18. [WARN] No validation on user-supplied name and email — `quiz.py` lines 153–165**

`create_account()` checks that `name` and `email` are not empty (lines 154–156 and 159–161), which is good. However, there is no format validation on the email (it could be `"x"` or `"<script>"`), and no length limits on name or password. For a file-backed terminal app this is low-risk, but excessively long inputs could bloat `users.dat`.

**19. [PASS] File paths use `os.path.join` with a fixed base directory — `quiz.py` lines 8–13**

All file paths are constructed as `os.path.join(BASE_DIR, "filename")` where `BASE_DIR` is derived from `os.path.dirname(os.path.abspath(__file__))`. No user input is ever interpolated into file paths, so path-traversal attacks are not possible.

**20. [PASS] No use of `eval()`, `exec()`, or `pickle` — entire file**

The codebase does not use `eval()`, `exec()`, `os.system()`, `subprocess`, or `pickle` anywhere. The `.dat` files use `zlib` + `json`, which cannot execute arbitrary code during deserialization. This eliminates the most common code-injection vectors in Python applications.

**21. [WARN] No file permission restrictions on sensitive data files — `quiz.py` lines 76–77**

`users.dat` and `scores.dat` are created with the OS default permissions. On shared systems, other users may be able to read (and potentially decompress) these files. Setting restrictive permissions (e.g., `0o600`) via `os.chmod()` after creation would limit access to the file owner only. For a single-user homework project this is low-risk.

### Overall Assessment

The codebase demonstrates strong security fundamentals for a homework-scale application. Passwords are hashed with a modern KDF (PBKDF2-HMAC-SHA256 with per-user salt and 100k iterations), `.dat` files are properly obfuscated via zlib-compressed JSON (avoiding pickle entirely), all file I/O uses context managers to prevent resource leaks, and no dangerous functions (`eval`, `exec`, `pickle`) are present. The six warnings are best-practice improvements — the legacy hash fallback and visible password echo being the most actionable. No critical security failures were found.
