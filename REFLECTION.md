I would map out like a visual diagram first and then clearly make headers subheaders and detail each of them out. I'd try to have much more elaborate picture in my head and explain really clearly in my spec. I might also try adding vague things as open to creativity for the agent to build on and see what it comes up with. 
When would you use this workflow? Based on this experience, when do you think plan-delegate-review is better than conversational back-and-forth? When is it worse?
I think when I have a clear flow and a realistic problem statement and use case for it, I think this workflow is really helpful. For my project 2 I tried to make tartan tuesday app which is a fixed problem to solution based task having a rigid use case, I could have benefitted with this workflow for that kind of project. Whereas, iterative projects which I have a raw idea about and want to see the first iteration first and then iteratively keep developing on it maybe a game where i need to improve visual aspects and graphics, I think it's not the best workflow to follow as the guidelines spec won't really make it clear inside my head how the graphic is going to look, I need to really see how the agent implements it. Maybe projects like that, more visual ones, require a different approach.
